# Search relevance eval harness

Offline harness that scores the **exact** production search pipeline
(`src/lib/search-core.ts`) against grounded query sets, so every relevance
change is measured, significance-tested, and ablated instead of guessed.

This corpus punishes intuition: prior rounds measured semantic synonym
expansion, `threshold:0`, `tolerance:2`, BM25 `b`/`k` retuning, and a
title-prefix bonus as **regressions or no-ops** despite all being "obvious"
wins. Nothing ships without a number ‚Äî and now, without a number on data the
tuning **never saw**.

## Why two eval sets (the overfitting problem)

`evalset.json` is 126 hand-written queries. A change that helps them can be
memorization, not relevance ‚Äî e.g. the best-bet `pins` literally hard-code
curated navigational queries, so they inflate curated metrics by construction.
To measure **generalization**, `mine-evalset.ts` auto-generates a large
held-out set from the corpus itself (page titles, `keywords:` frontmatter,
H2/H3 headings, path slugs, single-edit typos). Every mined query's target
URL is derived from the page's own structure and hard-validated against the
built index, then split **by target page** into train/test so a page's
title-query and a typo of it can't straddle the split.

The decision rule: a lever ships only if it is a **statistically significant,
positive move on the held-out mined-test slice** (paired permutation test,
10k resamples) **and** does not regress the curated set.

## Run

```bash
# from next/ ‚Äî all need a built index at out/api/search (npm run build)
npx tsx scripts/search-eval/run.ts              # legacy: baseline vs tuned + ablation
npx tsx scripts/search-eval/report.ts           # 3-slice matrix + significance + regressions
npx tsx scripts/search-eval/report.ts --quick   # baseline vs shipped tuned, 3 slices
npx tsx scripts/search-eval/report.ts --determinism
npx tsx scripts/search-eval/sweep.ts            # coordinate-ascent optimizer (held-out)
npx tsx scripts/search-eval/confirm.ts          # ship-candidate confirmation
npx tsx scripts/search-eval/mine-evalset.ts     # (re)generate the held-out set
INDEX=out/api/search npx tsx scripts/search-eval/report.ts

# also: npm run search:report | search:sweep | search:mine | search:eval
```

Query-side levers (stopwords, pins, spell, structHit, **bm25 blend, exact
title**) need no rebuild. Index-side levers (code symbols, `keywords`
frontmatter, tokenizer `allowDuplicates`) require `npm run build` because the
fumadocs index is produced by the webpack build.

## Pieces

- `evalset.json` ‚Äî 126 curated queries, 7 intents. Hard-validated (run fails
  loudly if an expected/pin URL is missing, so it can't silently rot).
- `mined-evalset.json` ‚Äî 1375 auto-generated, index-grounded, held-out
  queries (~12√ó curated), regenerated by `mine-evalset.ts`. Never overlaps
  curated. Split by target page into mined-train / mined-test.
- `lib/metrics.ts` ‚Äî Hit@1/5, Coverage@10, MRR, **nDCG@10, Recall@10,
  Precision@10, MAP**, all over distinct pages in rank order (what a user
  sees). Hit@1/MRR are the priority for docs search.
- `lib/stats.ts` ‚Äî bootstrap CI + paired permutation test on the delta. A
  +0.01 on 126 queries is often one query flipping; this is what tells signal
  from noise.
- `lib/split.ts` ‚Äî deterministic, page-stratified train/test split.
- `lib/harness.ts` ‚Äî shared evaluator; loads the index once, replays any
  `Tuning` through the real `runRankedSearch`.
- `report.ts` ‚Äî every variant on curated / mined-train / mined-test, with
  per-intent nDCG, significance vs the shipped tuning, and a per-query
  regression list (net-flat aggregates hide "won 4 / lost 4").
- `sweep.ts` ‚Äî coordinate ascent over the continuous levers, optimizing the
  objective on **mined-train only**, reporting the train‚Üîtest gap (overfit
  detector) and a held-out accept/reject verdict.
- `confirm.ts` ‚Äî measures the combined ship candidate (levers interact) on
  all slices with significance before it enters `DEFAULT_TUNING`.
- `run.ts` ‚Äî original 126-query ablation runner (kept; still green).
- `browser-path-check.ts` ‚Äî exercises the real client path end to end
  (fetch ‚Üí load ‚Üí rank ‚Üí highlight).

## LLM-augmented harness (Phases 2‚Äì7)

The 126-curated + 1375-mined slices measure binary relevance. To add
**realistic phrasing**, **multi-target** ground truth, **graded** relevance
(distinguishes perfect vs. acceptable), and a **diagnostic** failure
catalog, a sub-agent pipeline produces three more artifacts:

| stage              | model        | output                              | role                                                                                  |
| ------------------ | ------------ | ----------------------------------- | ------------------------------------------------------------------------------------- |
| Haiku generation   | Haiku 4.5    | `llm-candidates.jsonl` (4 649)      | 10 queries / page across 5 personas, 7 intents, 3 length buckets                      |
| Sonnet validation  | Sonnet 4.6   | `llm-validated.jsonl` (4 231)       | per-category adversarial audit: drop / expand / correct / keep ‚Äî 91% retention        |
| Opus calibration   | Opus 4.7     | `opus-calibration-report.json`      | **GATE**: graded-rank 126 curated, measure top-1 agreement / recall@expect            |
| Opus 3-session     | Opus 4.7 √ó3  | `gold-evalset.json` (349 queries)   | graded relevance, median across sessions, Krippendorff Œ± reported per query           |
| Opus red-team      | Opus 4.7 √ó3  | `hard-cases.json` (46)              | proposed failure queries, **empirically verified** against the pipeline; surviving 46 |

Phase-4 calibration result on curated: **top-1 agreement 0.9762** (123/126
match the hand label as the highest-grade page), **recall@expect 0.9921**
(125/126 have at least one expect URL graded ‚â• 2), false-3 rate 0.024.
Opus is a trustworthy graded judge on this corpus.

Phase-5 gold slice (stratified Opus 3-session ranking, full 350-query
sample): **Krippendorff Œ± median 1.000**, p10 = 0.964, max = 1.000 ‚Äî
3-session agreement is exceptional, 1 query dropped for low Œ± (kept 349).

### Graded metrics on the gold slice

`report-graded.ts` adds:

- **graded nDCG@10** with Burges gain `2^g ‚àí 1` (g ‚àà {0,1,2,3})
- **ERR@10** (Chapelle et al. cascade model: probability the user stops
  satisfied at rank r)
- **mean grade-at-rank-1** (out of 3 ‚Äî "did we put the BEST page first?")

Initial Phase-7 graded numbers (commit 5a28287, n=112) were directionally
positive but not significant. Subsequent rounds (headingMatchWeight,
allowDuplicates, description indexing, +keywords, spell additions,
queryNorm fix, shape-conditional code-symbol bonus) plus scaling the gold
slice to n=349 turned the direction into a significant move:

| metric         | baseline | shipped tuned | Œî        | p       |
| -------------- | -------- | ------------- | -------- | ------- |
| Hit@1 (binary) | 0.5014   | **0.5358**    | +0.0344  |         |
| MRR            | 0.5587   | **0.5900**    | +0.0312  | 0.004 ‚ñ≤ |
| nDCG-graded@10 | 0.4982   | **0.5134**    | +0.0152  | 0.081   |
| ERR@10         | 0.4506   | **0.4669**    | +0.0163  | 0.049 ‚ñ≤ |
| grade@1 mean   | 1.52     | **1.59**      | +0.0659  | 0.046 ‚ñ≤ |

3 of 4 graded metrics significantly improve on n=349 (paired permutation,
10k iters). nDCG_g@10 is directionally positive (p=0.08) ‚Äî the BM25 blend
+ code-symbol bonus move first-place outcomes more than they shift the
tail distribution.

**Per-intent diagnostic (nDCG_g@10, tuned)**: navigational 0.717,
exact 0.537, troubleshooting 0.520, typo 0.496, concept 0.473,
identifier 0.427, synonym 0.424. The remaining headroom is on
**synonym** and **concept** intents ‚Äî `identifier` improved with the
shape-conditional code-symbol bonus (0.408 ‚Üí 0.427 on this round).

### Run the LLM pipeline

The sub-agents are launched via the Agent tool from the orchestrator
session (no `ANTHROPIC_API_KEY` needed ‚Äî uses the Claude Max plan).
Each phase writes its task-manifest, the orchestrator dispatches
sub-agents that Read source files and Write the JSON outputs, then the
aggregator builds the final artifact:

```bash
# from next/, requires out/api/search and content/docs/
npx tsx scripts/search-eval/orchestrate-haiku.ts             # prepare batches
# (dispatch ~48 Haiku batches via Agent tool, model:haiku)
npx tsx scripts/search-eval/orchestrate-haiku.ts --aggregate # ‚Üí llm-candidates.jsonl

npx tsx scripts/search-eval/orchestrate-sonnet.ts            # prepare batches
# (dispatch ~65 Sonnet batches via Agent tool, model:sonnet)
npx tsx scripts/search-eval/orchestrate-sonnet.ts --aggregate # ‚Üí llm-validated.jsonl

npx tsx scripts/search-eval/orchestrate-opus-calibrate.ts          # prepare
# (dispatch 13 Opus calibration batches, model:opus)
npx tsx scripts/search-eval/orchestrate-opus-calibrate.ts --aggregate
# GATE: pass requires top1‚â•0.85 AND recall@expect‚â•0.95

npx tsx scripts/search-eval/orchestrate-opus-rank.ts         # prepare (350√ó3)
# (dispatch ~105 Opus rank batches, throttle to ‚â§10 parallel to avoid 529)
npx tsx scripts/search-eval/orchestrate-opus-rank.ts --aggregate # ‚Üí gold-evalset.json

npx tsx scripts/search-eval/orchestrate-opus-redteam.ts        # prepare 3 sessions
# (dispatch 3 Opus red-team sub-agents)
npx tsx scripts/search-eval/orchestrate-opus-redteam.ts --aggregate # ‚Üí hard-cases.json
# (each proposed failure is run through runRankedSearch ‚Äî only true failures
# survive, dropping Opus mispredictions)

# Final graded report:
npx tsx scripts/search-eval/report-graded.ts --vs-baseline
```
