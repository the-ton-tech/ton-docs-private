# Search relevance eval harness

Offline harness that scores the **exact** production search pipeline
(`src/lib/search-core.ts`) against grounded query sets, so every relevance
change is measured, significance-tested, and ablated instead of guessed.

This corpus punishes intuition: prior rounds measured semantic synonym
expansion, `threshold:0`, `tolerance:2`, BM25 `b`/`k` retuning, and a
title-prefix bonus as **regressions or no-ops** despite all being "obvious"
wins. Nothing ships without a number — and now, without a number on data the
tuning **never saw**.

## Why two eval sets (the overfitting problem)

`evalset.json` is 126 hand-written queries. A change that helps them can be
memorization, not relevance — e.g. the best-bet `pins` literally hard-code
curated navigational queries, so they inflate curated metrics by construction.
To measure **generalization**, `mine-evalset.ts` auto-generates a large
held-out set from the corpus itself (page titles, `keywords:` frontmatter,
H2/H3 headings, path slugs, single-edit typos). Every mined query's target
URL is derived from the page's own structure and hard-validated against the
built index, then split **by target page** into train/test so a page's
title-query and a typo of it can't straddle the split.

The decision rule: a lever ships only if it is a **statistically significant,
positive move on the held-out mined-test slice** (paired permutation test,
10k resamples) **and** does not regress the curated set.

## Run

```bash
# from next/ — all need a built index at out/api/search (npm run build)
npx tsx scripts/search-eval/run.ts              # legacy: baseline vs tuned + ablation
npx tsx scripts/search-eval/report.ts           # 3-slice matrix + significance + regressions
npx tsx scripts/search-eval/report.ts --quick   # baseline vs shipped tuned, 3 slices
npx tsx scripts/search-eval/report.ts --determinism
npx tsx scripts/search-eval/sweep.ts            # coordinate-ascent optimizer (held-out)
npx tsx scripts/search-eval/confirm.ts          # ship-candidate confirmation
npx tsx scripts/search-eval/mine-evalset.ts     # (re)generate the held-out set
INDEX=out/api/search npx tsx scripts/search-eval/report.ts

# also: npm run search:report | search:sweep | search:mine | search:eval
```

Query-side levers (stopwords, pins, spell, structHit, **bm25 blend, exact
title**) need no rebuild. Index-side levers (code symbols, `keywords`
frontmatter, tokenizer `allowDuplicates`) require `npm run build` because the
fumadocs index is produced by the webpack build.

## Pieces

- `evalset.json` — 126 curated queries, 7 intents. Hard-validated (run fails
  loudly if an expected/pin URL is missing, so it can't silently rot).
- `mined-evalset.json` — 1375 auto-generated, index-grounded, held-out
  queries (~12× curated), regenerated by `mine-evalset.ts`. Never overlaps
  curated. Split by target page into mined-train / mined-test.
- `lib/metrics.ts` — Hit@1/5, Coverage@10, MRR, **nDCG@10, Recall@10,
  Precision@10, MAP**, all over distinct pages in rank order (what a user
  sees). Hit@1/MRR are the priority for docs search.
- `lib/stats.ts` — bootstrap CI + paired permutation test on the delta. A
  +0.01 on 126 queries is often one query flipping; this is what tells signal
  from noise.
- `lib/split.ts` — deterministic, page-stratified train/test split.
- `lib/harness.ts` — shared evaluator; loads the index once, replays any
  `Tuning` through the real `runRankedSearch`.
- `report.ts` — every variant on curated / mined-train / mined-test, with
  per-intent nDCG, significance vs the shipped tuning, and a per-query
  regression list (net-flat aggregates hide "won 4 / lost 4").
- `sweep.ts` — coordinate ascent over the continuous levers, optimizing the
  objective on **mined-train only**, reporting the train↔test gap (overfit
  detector) and a held-out accept/reject verdict.
- `confirm.ts` — measures the combined ship candidate (levers interact) on
  all slices with significance before it enters `DEFAULT_TUNING`.
- `run.ts` — original 126-query ablation runner (kept; still green).
- `browser-path-check.ts` — exercises the real client path end to end
  (fetch → load → rank → highlight).

Per-intent gold nDCG_g@10:

| --------------- | -------- | -------- |

The biggest movement was on `exact`-intent (+0.293) and grade-at-rank-1

and the new `headingMatchWeight=0.2` lever.

### Lever verdicts (ablated, significance-tested)

| lever                             | verdict    | evidence                                                                                                                                                                                                                     |
| --------------------------------- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| domain-aware stopwords            | **keep**   | identifier 0.83→0.89, no regression                                                                                                                                                                                          |
| best-bet pins                     | **keep**   | navigational 0.61→1.00; also fire on spell-corrected query                                                                                                                                                                   |
| spell (always-union)              | **keep**   | typo 0.75→1.00; low-recall gate never fired (mis-ranking, not recall)                                                                                                                                                        |
| code-symbol + keyword indexing    | **keep**   | synonym 0.79→1.00, identifier +; index +2.5%                                                                                                                                                                                 |
| structHit (keyword rows only)     | **keep**   | code-symbol rows excluded — measured too noisy for NL queries (wrecked concept 1.00→0.88)                                                                                                                                    |
| **calibrated BM25 blend (w=2.5)** | **keep**   | the shipped pipeline _discarded Orama's BM25 score entirely_ (ranked by a coarse integer heuristic + crawl order). Re-introduced as `w·bm25/maxBm25`: held-out MRR/Hit@1/nDCG all +1.5–1.8pp, p≤0.0004; 0 curated regression |
| **exact-title bonus (w=3)**       | **keep**   | substring `includes` ranked "Wallet" = "How wallets work"; exact/prefix preference: independent held-out significant gain, additive with BM25 blend, 0 curated regression                                                    |
| BM25 blend w=3                    | **reject** | +14% more held-out gain but regresses 2 curated queries; curated is the higher-confidence signal — w=2.5 is the Pareto knee                                                                                                  |
| `relevance` BM25 k/b/d retune     | **reject** | every off-default value net-negative on held-out, both directions (b=0.9 sent "block explorer" 1→50)                                                                                                                         |
| title-prefix bonus                | **reject** | not significant on held-out (p≈0.06–0.12); off for parsimony                                                                                                                                                                 |
| proximity / all-terms bonus       | **reject** | nets −Hit@1/−MRR (floats long reference pages over canonical); measured 5×                                                                                                                                                   |

Residual 2 curated fails are code-only-symbol cases (`OP_SENDMSG`,
`loadUint`). Accepted — chasing them risks the overfitting this harness
exists to prevent, now quantified by the held-out slice.

## LLM-augmented harness (Phases 2–7)

The 126-curated + 1375-mined slices measure binary relevance. To add
**realistic phrasing**, **multi-target** ground truth, **graded** relevance
(distinguishes perfect vs. acceptable), and a **diagnostic** failure
catalog, a sub-agent pipeline produces three more artifacts:

| stage              | model        | output                              | role                                                                                  |
| ------------------ | ------------ | ----------------------------------- | ------------------------------------------------------------------------------------- |
| Haiku generation   | Haiku 4.5    | `llm-candidates.jsonl` (4 649)      | 10 queries / page across 5 personas, 7 intents, 3 length buckets                      |
| Sonnet validation  | Sonnet 4.6   | `llm-validated.jsonl` (4 231)       | per-category adversarial audit: drop / expand / correct / keep — 91% retention        |
| Opus calibration   | Opus 4.7     | `opus-calibration-report.json`      | **GATE**: graded-rank 126 curated, measure top-1 agreement / recall@expect            |
| Opus 3-session     | Opus 4.7 ×3  | `gold-evalset.json` (112 queries)   | graded relevance, median across sessions, Krippendorff α reported per query           |
| Opus red-team      | Opus 4.7 ×3  | `hard-cases.json` (46)              | proposed failure queries, **empirically verified** against the pipeline; surviving 46 |

Phase-4 calibration result on curated: **top-1 agreement 0.9762** (123/126
match the hand label as the highest-grade page), **recall@expect 0.9921**
(125/126 have at least one expect URL graded ≥ 2), false-3 rate 0.024.
Opus is a trustworthy graded judge on this corpus.

Phase-5 gold slice (stratified Opus 3-session ranking): **Krippendorff α
median 1.000**, p10 = 0.963, min = 0.823 — 75%+ of queries have perfect
3-session agreement, 0 dropped for low α. Sample is 112 queries (partial
of 350 planned; rate-limit gated, fully resumable).

### Graded metrics on the gold slice

`report-graded.ts` adds:

- **graded nDCG@10** with Burges gain `2^g − 1` (g ∈ {0,1,2,3})
- **ERR@10** (Chapelle et al. cascade model: probability the user stops
  satisfied at rank r)
- **mean grade-at-rank-1** (out of 3 — "did we put the BEST page first?")

Initial Phase-7 graded numbers (commit 5a28287, before the headingMatch /
keywords / queryNorm-fix rounds) — direction positive but not
significant on n=112 (p ≥ 0.55), exact-intent grade@1 mean stuck at 1.0:

| metric         | baseline | shipped @ 5a28287 |
| -------------- | -------- | ----------------- |
| Hit@1 (binary) | 0.634    | 0.652             |
| MRR            | 0.692    | 0.702             |
| nDCG-graded@10 | 0.593    | 0.586             |
| ERR@10         | 0.564    | 0.563             |
| grade@1 mean   | 1.90     | 1.92              |

Subsequent rounds (headingMatchWeight, allowDuplicates, description
indexing, +keywords, spell additions, queryNorm fix) turned the
direction into a significant move — see top-of-doc "Validated result"
table for current shipped numbers.

- `lib/personas.ts` — 5 user personas (novice/expert/non_native/
  troubleshooter/explorer) for Haiku prompt diversity.
- `lib/llm-prompts.ts` — byte-stable prompt builders for every phase
  (Haiku gen, Sonnet validation, Opus graded-rank, Opus red-team).
- `lib/llm-types.ts` + `lib/llm-validate.ts` — zod schemas + tolerant

  final consolidated artifacts.
- `lib/pages.ts` — shared corpus loader (URL = `/` + path under
  `content/docs/` sans `.mdx`; verified mapping).
- `orchestrate-haiku.ts` / `orchestrate-sonnet.ts` /
  `orchestrate-opus-calibrate.ts` / `orchestrate-opus-rank.ts` /
  `orchestrate-opus-redteam.ts` — task-manifest builders + aggregators
  (all resumable: skip outputs that already exist).
- `report-graded.ts` — Phase-7 graded-metric report on the gold slice.
- `smoke-test.ts` — Phase-1 infrastructure verifier (24 checks; no LLM).

### Run the LLM pipeline

The sub-agents are launched via the Agent tool from the orchestrator
session (no `ANTHROPIC_API_KEY` needed — uses the Claude Max plan).
Each phase writes its task-manifest, the orchestrator dispatches
sub-agents that Read source files and Write the JSON outputs, then the
aggregator builds the final artifact:

```bash
# from next/, requires out/api/search and content/docs/
npx tsx scripts/search-eval/orchestrate-haiku.ts             # prepare batches
# (dispatch ~48 Haiku batches via Agent tool, model:haiku)
npx tsx scripts/search-eval/orchestrate-haiku.ts --aggregate # → llm-candidates.jsonl

npx tsx scripts/search-eval/orchestrate-sonnet.ts            # prepare batches
# (dispatch ~65 Sonnet batches via Agent tool, model:sonnet)
npx tsx scripts/search-eval/orchestrate-sonnet.ts --aggregate # → llm-validated.jsonl

npx tsx scripts/search-eval/orchestrate-opus-calibrate.ts          # prepare
# (dispatch 13 Opus calibration batches, model:opus)
npx tsx scripts/search-eval/orchestrate-opus-calibrate.ts --aggregate
# GATE: pass requires top1≥0.85 AND recall@expect≥0.95

npx tsx scripts/search-eval/orchestrate-opus-rank.ts         # prepare (350×3)
# (dispatch ~105 Opus rank batches, throttle to ≤10 parallel to avoid 529)
npx tsx scripts/search-eval/orchestrate-opus-rank.ts --aggregate # → gold-evalset.json

npx tsx scripts/search-eval/orchestrate-opus-redteam.ts        # prepare 3 sessions
# (dispatch 3 Opus red-team sub-agents)
npx tsx scripts/search-eval/orchestrate-opus-redteam.ts --aggregate # → hard-cases.json
# (each proposed failure is run through runRankedSearch — only true failures
# survive, dropping Opus mispredictions)

# Final graded report:
npx tsx scripts/search-eval/report-graded.ts --vs-baseline
```
