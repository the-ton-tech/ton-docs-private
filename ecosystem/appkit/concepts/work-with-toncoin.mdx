Toncoin is the native currency of the TON blockchain. AppKit exposes Toncoin through balance reads, transfer actions, and the settlement trace API.

## Balance

Balances are returned as strings representing fractional Toncoin units. For example, `'0.1'` is a balance of 0.1 Toncoin, or 100000000 nanoToncoin. Decimal precision is 9 digits, so the smallest possible balance is `'0.000000001'` Toncoin or 1 nanoToncoin.

Blockchain state changes constantly. A discrete one-off balance check has almost no value on its own — the state may change immediately after the query completes, invalidating the result. Use one-off checks only when assembling an outgoing transfer; use continuous monitoring (streaming) for UI display.

See [How to > Read balances](/ecosystem/appkit/howto/read-balances) for the read code and [Get Started > Streaming](/ecosystem/appkit/get-started/streaming) for live updates.

## Transfers

A Toncoin transfer carries a recipient address, an amount in fractional Toncoin, and an optional comment or base64 payload. AppKit assembles the request; the wallet signs and broadcasts; the chain settles.

See [How to > Send Toncoin](/ecosystem/appkit/howto/send-toncoin) for the component, hook, and core action.

## Finality

TON achieves transaction [finality](https://en.wikipedia.org/wiki/Blockchain#Finality) after a single masterchain block confirmation, where new blocks are produced approximately every 3 seconds. Once a transaction appears in a masterchain block, it becomes irreversible.

`sendTransaction` returning a `boc` means **the user signed**. Settlement is a separate state — verify with `getTransactionStatus(appKit, { boc })`, which returns `'unknown' | 'pending' | 'completed' | 'failed'` along with `totalMessages`, `pendingMessages`, and `onchainMessages` for trace progress. Treat only `status === 'completed'` as final.

Applications should not block the UI while waiting for confirmation. Show a "signed / confirming" state, let streaming push live updates, and act on `completed` when it arrives. For finding a transaction by external message hash and applying normalization, see the [message lookup guide](/ecosystem/ton-connect/message-lookup).

## Rules

- Validate the recipient address, amount, and expected network before calling any mutation. Do not rely on the wallet as the final input validator.
- Treat `sendTransaction` returning a `boc` as "user signed", not "settled". Wait for `getTransactionStatus` to return `completed` before delivering value.
- Do not block UI while waiting for settlement. Show a "signed / confirming" state, let `watchBalance` / `watchTransactions` update the view, and act on `completed` when it arrives.
- Refetch wallet balances and cached lists after settlement. Use trace status, not the immediate wallet response, as the trigger for product state changes.
- Do not derive product success from the wallet response shape. Verify on the chain.
- Keep amounts as strings until you display them. Floating-point math loses precision at TON's nine decimal places.
