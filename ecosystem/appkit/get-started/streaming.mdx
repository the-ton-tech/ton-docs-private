Streaming is the live channel of AppKit: a per-network subscription layer that turns provider events into balance, jetton, and transaction updates pushed to the app. Use it instead of polling whenever a UI needs to reflect chain changes without a user-driven refresh. See [Streaming concept](/ecosystem/appkit/concepts/streaming) for the model and the `update.status` lifecycle.

## Before you begin

You need a streaming provider registered on the AppKit instance before any watcher will fire. See [Providers → Streaming providers](/ecosystem/appkit/get-started/providers#provider-types) for the registration pattern, and use `hasStreamingProvider(appKit, network)` from `@ton/appkit` as the runtime check before subscribing — the check is per-network, so pass the network you intend to watch.

## Mount global watchers

Mount `useWatch*` hooks once at a high level, such as the router, app shell, or layout. Calling a hook with no arguments keeps the cache warm. Pass `onChange` only for side effects such as toasts, logs, or navigation.

```tsx
import { useWatchBalance, useWatchTransactions } from '@ton/appkit-react';

export function GlobalWatchers() {
  useWatchBalance();
  useWatchTransactions({
    onChange: (update) => {
      if (update.status === 'finalized') {
        console.log('settled', update.traceHash);
      }
    },
  });
  return null;
}
```

`update.status` is `'pending' | 'confirmed' | 'finalized' | 'invalidated'`. Treat anything before `'finalized'` as not yet settled. See [Streaming concept → Transaction status lifecycle](/ecosystem/appkit/concepts/streaming#transaction-status-lifecycle) for what each status means and when product-state changes are safe.

## Rule

A stream is a hint, not proof. Verify the final state on a backend before delivering value or settling an order, and re-check after every reconnect — the stream may have missed transitions while disconnected.
